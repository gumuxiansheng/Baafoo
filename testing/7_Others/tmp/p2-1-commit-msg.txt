feat(core): P2-1 introduce MatchRequest to replace 14-parameter match methods

Adds MatchRequest — a fluent POJO encapsulating the 11-14 match inputs
(protocol, host, port, serviceName, method, path, topic, headers,
queryParams, body, grpcService, grpcMethod) that were previously passed
positionally to MatchEngine.match() / matchWithFallback().

Benefits:
- Readable call sites (named parameters via fluent setters).
- Prevents argument-order bugs (e.g., swapping path and topic).
- Allows adding new match dimensions without changing every call site.

New overloads added to MatchEngine:
- match(List<Rule>, MatchRequest)
- matchWithFallback(List<Rule>, MatchRequest) — restores the original
  port on the request after the port=0 retry, so callers see no mutation.

The old positional overloads remain for backward compatibility; the new
MatchRequest overloads are preferred for new call sites. Existing
handlers and tests are not migrated in this commit — that follows per
handler in subsequent commits.

Factory methods:
- MatchRequest.http(...) for HTTP-style protocols
- MatchRequest.mq(...) for Kafka/Pulsar/JMS
- MatchRequest.grpc(...) for gRPC-over-HTTP/2
- MatchRequest.tcp(...) for raw TCP

Two new tests verify equivalence with the positional overloads and the
port=0 fallback behavior.
