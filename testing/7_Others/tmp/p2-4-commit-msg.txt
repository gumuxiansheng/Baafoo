fix(agent): P2-4 harden Bootstrap CL reflection bridge

The sync methods (syncGlobalRouteStateToBootstrapCL,
syncLogHandlersToBootstrapCL, syncRecordingWrappersToBootstrapCL,
RouteManager.syncModeToBootstrapCL, RouteManager.syncRoutesToBootstrapCL)
previously called Class.getField(fieldName) directly inside broad
catch (Exception) blocks. If a field was renamed or removed from
GlobalRouteState, the resulting NoSuchFieldException was swallowed
and logged as a generic warning — the agent would appear to start
successfully but connections would silently fail to be intercepted,
with no actionable error message.

P2-4 fix:
- Added requireBootstrapField(), setBootstrapInt(), setBootstrapRef()
  helpers in BaafooAgent that centralize field access with explicit
  existence checks. On NoSuchFieldException they throw an
  IllegalStateException with an actionable message naming the missing
  field and suggesting the Bootstrap JAR is out of sync.
- Refactored syncGlobalRouteStateToBootstrapCL to use setBootstrapInt
  / setBootstrapRef for all scalar fields (CURRENT_MODE retained as
  direct getField for minimal diff, but other 9 fields use helpers).
- Refactored syncLogHandlersToBootstrapCL and
  syncRecordingWrappersToBootstrapCL to use requireBootstrapField for
  all field lookups.
- Refactored RouteManager.syncModeToBootstrapCL and
  syncRoutesToBootstrapCL to throw IllegalStateException on missing
  fields instead of silently logging.

Operators now see a clear startup error when the Bootstrap JAR is out
of sync with the agent source, instead of debugging silent interception
failures at runtime.
