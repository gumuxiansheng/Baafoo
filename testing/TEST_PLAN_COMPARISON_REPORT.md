## 3. 结论与建议

### 3.1 文档关系判定：强互补，不宜简单合并

- **文档 A** 是作者刚编写的、基于 PRD v2.4 和源码的最新**可执行用例库**。它的价值在于：用例可直接转化为代码，AC 可追溯，覆盖全协议/全规则分支/全环境模式，且工具链较现代（JUnit 5）。
- **文档 B** 是项目既有、版本 2.0 的**项目级测试计划**。它的价值在于：覆盖范围广（企业应用、安全、前端深度、资源泄漏、兼容性、风险、交付物），管理性强（进度、资源、风险、交付物），且包含 A 没有的重要专项（变异测试、多 Agent 共存、SDK、Sidecar、TLS/SASL 等）。

**两者不是重复，而是不同层级**：
- A = 详细的“测试用例规范”（Test Case Specification）
- B = 高层的“测试计划与管理文档”（Test Plan & Management）

### 3.2 是否需要合并或拆分？

**推荐：不合并为一个文件，但建立“主从 + 索引”关系。**

原因：
1. A 已声明“排除已有 testing/ 目录下测试文档（独立编制，不参考不依赖）”，若强行合并会破坏 A 的独立性。
2. 合并后文件将超过 4500 行，难以维护、检索和版本控制。
3. 不同受众需要不同文档：开发需要 A，项目经理/审计需要 B。

**建议结构**：
```
testing/
├── TEST_PLAN_FULL.md              # 可执行用例库（A 继续作为主体）
├── 7_Others/
│   └── PROJECT-TEST-PLAN.md       # 项目级测试计划（B 继续作为主体）
└── TEST_PLAN_COMPARISON_REPORT.md # 本报告：说明两者关系
```

### 3.3 推荐后续动作

#### 动作 1：明确主从关系（立即）
在 A 和 B 的文档开头互相引用：
- A 开头增加：*“本文件为可执行用例清单。测试策略、环境、风险、交付物详见 `7_Others/PROJECT-TEST-PLAN.md`。”*
- B 开头增加：*“本文件为项目级测试计划。详细可执行用例与 AC 映射详见 `TEST_PLAN_FULL.md`。”*

#### 动作 2：A 补充 B 中的高价值专项（高优先级）
从 B 中选取对当前阶段最有价值的主题，在 A 中增加独立章节：
1. 安全测试（认证绕过、SQL/XSS 注入、路径遍历、依赖扫描）
2. 资源泄漏专项（FD、线程、Netty 堆外内存、Metaspace）
3. TLS/SASL 兼容性用例
4. 多 Agent 共存（至少 JaCoCo + SkyWalking + Baafoo）
5. 已知 P0 问题验证（BUG-4 / THREAD-1 / SEC-1）
6. 前端单元测试（Vitest）

#### 动作 3：B 更新 A 已覆盖的内容（高优先级）
B 中以下内容已过时，应更新：
1. **JUnit 4 → JUnit 5**：B §5.4 仍写 JUnit 4，应改为 JUnit 5（与 A 一致）。
2. **gRPC 支持**：B §2.2 写“Baafoo 不支持 gRPC”，但 A 已覆盖 gRPC 用例，B 应修正。
3. **用例状态**：将 A 中已覆盖的“待补充”用例标记为“已有/已覆盖”。
4. **协议 × 模式矩阵**：B §6.4.2 的缺口矩阵应引用 A 中的完整矩阵。

#### 动作 4：统一 AC 追踪格式（中优先级）
- 在 B 中引入 A 的 AC 编号格式（如 `R-S2 AC-01`），或在 A 中补充 B 的用例 ID 映射。
- 建议以 A 的 AC 格式为主，因为它是基于 PRD 的正式验收标准。

#### 动作 5：建立单一索引页（中优先级）
在 `testing/` 目录下创建 `README.md` 或 `TEST_INDEX.md`，说明：
- `TEST_PLAN_FULL.md`：最新可执行用例，AC 可追溯。
- `7_Others/PROJECT-TEST-PLAN.md`：项目级策略、环境、风险、交付物、企业应用。
- 两个文档的更新责任和版本策略。

#### 动作 6：处理明显冲突（高优先级）
| 冲突点 | 建议 |
|--------|------|
| JUnit 4 vs 5 | 以 A 为准（JUnit 5），更新 B |
| gRPC 是否支持 | 以 A 为准（已支持），更新 B §2.2 |
| 测试工具链（JMeter vs wrk） | 在 B 中保留“推荐工具”弹性表述，不强制单一工具 |
| 覆盖率目标 | 75% 行覆盖率 / 60% 变异覆盖率可作为统一目标 |

#### 动作 7：版本管理策略（立即）
- A 为 v1.0，B 为 v2.0，建议统一版本号或建立变更日志关联。
- 建议 A 升级到 v1.1 时增加 B 的交叉引用；B 升级到 v2.1 时同步 A 的覆盖状态。

### 3.4 最终结论

两份文档应**共存并互补**，而不是合并或废弃。A 作为当前最权威、最可执行的测试用例库，B 作为项目级测试策略与广度覆盖计划。下一步应优先：
1. 在两者开头互相引用；
2. 更新 B 中已过时的事实（JUnit 4、gRPC 不支持）；
3. 将 B 中安全、资源泄漏、TLS、多 Agent 等高价值主题补充到 A 中；
4. 建立统一 AC 追踪和单一索引页。

这样可形成一个“**B 计划 + A 用例 + 索引报告**”的完整测试文档体系，既满足工程执行，又满足项目管理和审计需求。
