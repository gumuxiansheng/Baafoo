feat(handler,agent): P1-1 introduce ModeStrategy abstraction

Consolidates the duplicated 5-way environment-mode decision trees that
appeared in HttpStubHandler, GrpcStubHandler, GrpcUnifiedHandler,
TcpStubHandler, and the MQ brokers, plus the duplicated mode-ordinal
checks scattered across the Bootstrap-CL socket advice classes.

Three-tier split respecting the classloader boundaries documented in
AGENTS.md:

1. baafoo-server (App CL): ModeStrategy + ModeOutcome + DefaultModeStrategy
   - ModeOutcome is a pure-data record describing four orthogonal
     decisions: forwardToDownstream, recordRequest, sendStub,
     recordUnmatched, recordResponse.
   - DefaultModeStrategy.INSTANCE resolves (mode, matched) → ModeOutcome
     for all five EnvironmentMode values.
   - Handlers retain full control of I/O; the strategy only decides
     *which* operations to perform.
   - 11 unit tests in DefaultModeStrategyTest cover the full matrix.

2. baafoo-agent App-CL advice: ModeGates.shouldIntercept(EnvironmentMode)
   - Static helper for the identical 4-line PASSTHROUGH early-return
     idiom in Kafka/JMS/Pulsar/gRPC advice classes. A full Strategy
     is over-engineering here because these advice classes defer the
     record-vs-stub distinction to the server-side handler.

3. baafoo-agent Bootstrap-CL advice: GlobalRouteState static int helpers
   - shouldIntercept(int), shouldRecordStream(int),
     isRecordAndStub(int), shouldRedirectUnmatched(int).
   - These live on GlobalRouteState so ByteBuddy-inlined bytecode in
     JDK classes (Socket, SocketChannel) can reach them without
     referencing the relocated baafoo-core EnvironmentMode enum.
   - A polymorphic Strategy is deliberately NOT used on the Bootstrap
     CL path because virtual dispatch + cross-classloader references
     risk NoClassDefFoundError at inlined call sites.

No handler is migrated to the new abstraction in this commit — that
follows in subsequent commits per handler. This commit only adds the
abstraction and its tests.
